# QuestAdmin 保存I/O方針

## 現在の保存方式

QuestAdmin は現在、`config/questadmin/` 配下のJSONファイルへ保存します。

- `quests.json`: クエスト定義
- `player_quests.json`: プレイヤーごとの進行状況と報酬受け取り状態

保存は同期処理です。一時ファイルへ書き込んでから対象ファイルを置換します。`ATOMIC_MOVE` が使えない環境ではwarnログを出し、非atomic置換へfallbackします。

Phase 11.6 では、以下の軽量な保存I/O計測を追加しています。

- 最後の保存時間ms
- 保存成功回数
- 保存失敗回数
- atomic fallback回数
- 50ms以上の遅い保存warn
- 200ms以上の強い遅い保存warn

これらは観測用の計測です。保存順序を変えたり、非同期保存へ変更したりはしていません。

## complete / claim を同期保存のまま維持する理由

`/quest complete <questId>` は納品アイテムを消費し、その後 `COMPLETED` 状態を保存します。保存に失敗した場合はインベントリsnapshotを戻して失敗を返します。この判断は保存結果を待つ必要があるため、同期保存を維持します。

`/quest claim <questId>` はさらに慎重に扱う必要があります。Lightman's Currency へ入金する前に `CLAIMING` を保存し、入金成功後にのみ `CLAIMED` を保存します。

現在の順序:

1. `CLAIMING` を保存する
2. Lightman's Currency へ報酬を入金する
3. `CLAIMED` を保存する

`CLAIMING` 保存前に入金してはいけません。また、報酬支払い後に `CLAIMED` 保存が失敗すると、二重支払いや状態不整合の危険があります。そのため、claim系処理は安易に非同期化しません。

`/questadmin progress mark ...` も管理者による状態修正コマンドです。管理者には実際の保存結果を返す必要があるため、同期保存を維持します。

## 無計画な非同期保存が危険な理由

非同期保存キューは、設計しないまま導入すると重要な状態遷移を遅延・並び替えする可能性があります。

起こり得る事故:

- アイテムを消費したのに `COMPLETED` が永続化されない
- 報酬を支払ったのに `CLAIMED` が永続化されない
- `CLAIMING` と `CLAIMED` の保存順序が崩れる
- サーバークラッシュ前に保存キューがflushされない
- 保存待ち中に `/quest claim` が再実行される

特に報酬支払い後の保存失敗は危険です。Lightman's Currency への入金が成功したあとに保存が失敗した場合、QuestAdminは管理者確認が必要な状態として扱います。単純なバックグラウンド保存では、この失敗をプレイヤーコマンドの完了後まで隠してしまう可能性があります。

将来非同期化する場合は、トランザクション状態、保存再試行キュー、クラッシュ復旧方針、管理者向けの可視化が必要です。

## 将来非同期化を検討しやすい候補

設計後に候補になり得るもの:

- 管理者のクエスト作成/編集/enabled切替による `quests.json` 保存
- 将来追加されるかもしれない低リスクな表示cacheや統計ファイル
- 報酬支払い・アイテム消費を伴わない管理者向け一括メンテナンス

debounce保存を検討できる可能性があるもの:

- 確定前のクエスト編集途中データ
- 将来追加される非重要なUI設定や表示cache
- 明示的な最終保存確認を持つ管理者一括操作

debounce保存してはいけないもの:

- `/quest complete <questId>`
- `/quest claim <questId>`
- `CLAIMING` 保存
- `CLAIMED` 保存
- Lightman's Currency 入金に隣接する保存
- `/questadmin progress mark ...`

## サーバー停止時flush

将来のPhaseでサーバー停止時flushを検討できます。ただし、Phase 11.6 では本格的なflushシステムは実装しません。

非同期保存やdebounce保存を導入する場合、サーバー停止時には必要な永続化を待ち、失敗をログへ明示し、未保存の進行状況を静かに失わない設計が必要です。

## 50人以上同時接続時の注意

50人以上の同時接続では、`player_quests.json` が大きくなるほど同期JSON保存がメインスレッドの遅延要因になります。Phase 11.6 では、このリスクを観測するために遅い保存ログと `/questadmin storage status` を追加しています。

運用上の注意:

- 50ms / 200ms の保存warnを監視する
- `player_quests.json` の肥大化に注意する
- 手動JSON編集前にはバックアップを取る
- ピーク時間帯の大量管理者編集を避ける
- claim保存失敗は報酬支払い絡みのインシデントとして扱う

トランザクションを意識した保存設計が入るまでは、書き込み性能よりも二重支払い防止と状態整合性を優先します。
